## 1. 概要・背景

### プロダクト名

街路樹現場調査ツール（Field Survey Tool）

### 目的

樹木医による街路樹診断業務（街路樹診断カルテ作成）を効率化する。
1日40〜50本の調査を行う樹木医が、現場でiPhoneから入力 → PCで自動的にカルテExcelを生成するワークフローを実現する。

### 開発者

樹木医本人（街路樹診断士）。実務での痛点ベースで開発・改善を回している。

### 競合・類似サービス

「街路樹note」（既存サービス）。差別化方針：
- ひとり樹木医のための、爆速カルテ作成ツール
- オフライン完結（電波の弱い現場でも動く）
- データはユーザー所有（ベンダーロックインなし）
- 樹木医本人が作っていることによる業務適合性

## 2. 業務的前提知識

エンジニア向け：街路樹診断業務の最低限の前提。

### 街路樹診断とは

道路沿いに植えられた樹木の健康状態・倒木リスクを評価する業務。樹木医（国家資格）が行う。
診断結果は「街路樹診断カルテ」というExcel形式の報告書として行政に提出される。

### 診断の3軸

樹木1本につき、以下の3軸で評価する：

1. **活力判定**：樹勢（生命力）と樹形（樹冠の形）の評価
2. **部位判定**：根元・幹・大枝それぞれの状態
3. **外観診断判定**：上記を踏まえた総合的な外観評価

各判定はA/B1/B2/Cの4段階：
- A：健全
- B1：軽度の損傷
- B2：中程度の損傷
- C：重度の損傷（要処置）

### 観察される代表的な異常

- **キノコ類（子実体）**：ベッコウタケ、コフキタケ等。木材腐朽菌の証拠
- **空洞**：開口空洞（外から見える）、芯達空洞（内部）
- **キャンカー（がんしゅ病）**
- **樹皮枯死・欠損・腐朽**
- **不自然な傾斜**：根元の異常
- **枯れ枝**：大枝レベルの枯死
- **葉量減少・黄化**：活力低下の兆候

### 1本あたりの作業時間

現場：5〜10分／本（写真撮影含む）
カルテ作成：従来は1本15〜30分。本ツールで5分以下を目指す。

## 3. システム全体像

### アーキテクチャ

```
┌─────────────────────┐    JSON     ┌──────────────────────┐
│  PWA (Field Survey) │ ──────────> │  Karte Generator     │
│   - iPhone (現場)   │   + 写真    │   - PC (Python)      │
│   - オフライン対応  │             │   - openpyxl + PIL   │
└─────────────────────┘             └──────────────────────┘
         │                                    │
         ▼                                    ▼
   IndexedDB                            カルテExcel
                                        （行政提出用）
```

### 2層構成にした理由

最初はPWAで完結を目指したが、ExcelJSはテンプレートExcelのスタイル定義（styles.xml）を破壊する致命的な問題があったため、Python+openpyxlに分離した。openpyxlの`copy_worksheet`は完璧にスタイルを保持できる。

### データフロー

1. **入力（PWA）**：樹ごとに基本情報・写真・診断データを入力 → IndexedDBに保存
2. **エクスポート（PWA）**：JSONとして出力。写真はZIPで同梱
3. **生成（PC）**：JSONをカルテ生成ツールに渡す（ドラッグ&ドロップ or GUI）
4. **テンプレート展開**：openpyxlでテンプレートExcelをコピー
5. **値貼り付け**：JSON内容を所定セルに貼り付け
6. **写真処理**：PILで全景写真にアノテーション（病害マーク）を焼き込み → Excelに貼付
7. **保存**：完成したカルテExcelを出力

## 4. データモデル

### 樹木データ（Tree）

PWAで管理される1本の樹木のデータ構造：

```typescript
{
  id: string,                  // 樹の一意ID
  treeNumber: string,          // 樹木番号（業務上の識別子）
  species: string,             // 樹種（例：「アカガシ」）
  height: number,              // 樹高 (m)
  trunkCircumference: number,  // 幹周 (cm)
  branchSpread: number,        // 枝張 (m)
  plantingForm: string,        // 植栽形態
  hasSupport: boolean,         // 支柱の有無
  
  vitalitySei: 1 | 2 | 3 | 4 | 5,  // 樹勢
  vitalityKei: 1 | 2 | 3 | 4 | 5,  // 樹形
  
  vitalityJudgment: 'A' | 'B1' | 'B2' | 'C',
  partJudgments: {
    root: 'A' | 'B1' | 'B2' | 'C',
    trunk: 'A' | 'B1' | 'B2' | 'C',
    branch: 'A' | 'B1' | 'B2' | 'C',
  },
  appearanceJudgment: 'A' | 'B1' | 'B2' | 'C',
  
  threeChoiceJudgments: {      // v2.6: 9マス x 3択
    root:   { barkDeath, cavityShallow, cavityDeep },
    trunk:  { ... },
    branch: { ... },
  },
  
  memo: string,                // 現場メモ（チップ入力で蓄積）
  
  vitalityReason: string,      // v2.7: 活力判定理由
  appearanceReason: string,    // v2.7: 外観診断判定理由
  specialNotes: string,        // v2.7: 特記事項
  
  photoFull: {
    blob: Blob,
    label: '全景',
    annotations: [             // v2.8: 病害マーク
      {
        id: string,
        anchorX: number,       // 0-1正規化座標（患部）
        anchorY: number,
        labelX: number,        // 0-1正規化座標（テキストボックス）
        labelY: number,
        text: string,
      }
    ]
  },
  photoCloseup1: { blob, label },  // クローズアップ写真1〜3
  photoCloseup2: { ... },
  photoCloseup3: { ... },
}
```

### 現場メモのフォーマット

チップタップで蓄積されるメモは以下の形式：

```
根元：不自然な傾斜、ベッコウタケ
幹：開口空洞
大枝：枯れ枝
（その他自由記述）
```

このフォーマットを `generateAppearanceReason()` がパースして部位別の文章を生成する。

### JSON エクスポート構造

```json
{
  "exportedAt": "2026-05-09T10:30:00Z",
  "trees": [ /* Tree[] */ ],
  "meta": {
    "siteName": "渋谷氷川の杜",
    "surveyDate": "2026-05-08",
    /* ... */
  }
}
```

写真の `blob` は JSON には含まれない。写真は ZIP の別ファイルとして同梱される（`photos/{treeNumber}_full.jpg` 等）。

## 5. PWA機能仕様

### 5.1 画面構成

単一画面（SPA）。樹木リストと現在表示中の樹の詳細を切り替える。

```
┌─────────────────────────────────────┐
│ ヘッダー（タイトル / エクスポート）  │
├─────────────────────────────────────┤
│ 樹タブ：[1 アカガシ] [+ 追加]        │
├─────────────────────────────────────┤
│                                     │
│ 基本情報セクション                   │
│  - 樹木番号、樹種、樹高...           │
│                                     │
│ 写真セクション                       │
│  - 全景（マーク機能あり）            │
│  - クローズアップ1, 2, 3             │
│                                     │
│ 活力判定セクション                   │
│  - 樹勢/樹形（1-5）                  │
│  - A/B1/B2/C                         │
│  - 判定理由 + ✨診断文生成ボタン      │
│                                     │
│ 部位判定セクション                   │
│  - 根元/幹/大枝 × A/B1/B2/C マトリクス│
│                                     │
│ 外観診断判定セクション               │
│  - A/B1/B2/C                         │
│  - 判定理由 + ✨診断文生成ボタン      │
│                                     │
│ 特記事項セクション                   │
│                                     │
│ 診断チップセクション                 │
│  - 部位別タップで現場メモに挿入      │
│                                     │
└─────────────────────────────────────┘
```

### 5.2 主要機能

#### a) 写真アップロード・圧縮

- 4枠（全景、クローズアップ1〜3）
- 撮影 or ライブラリ選択
- 自動圧縮：長辺960px、JPEG品質60%
- ラベル付け可能

#### b) 診断チップ（v2.5）

部位別（根元・幹・大枝）に診断項目（ベッコウタケ、開口空洞等）のチップが並ぶ。
タップすると現場メモに「部位：項目」形式で挿入される。
同部位は読点で連続、別部位は改行＋部位ラベルで整形される。

#### c) 三択選択（v2.6）

樹皮枯死、開口空洞、芯達空洞 × 根元・幹・大枝 = 9マスを「なし／あり／重度」の3択で選ぶ折りたたみセクション。

#### d) 診断文生成（v2.7、v2.7.1）

「✨ 診断文生成」ボタンで活力／外観の判定理由を自動生成。

**生成ロジック**：
- 辞書ベース（オンライン依存なし）
- 樹勢/樹形の数値、メモのキーワード抽出、外観判定の値を組み合わせて樹木医のカルテ調文章を生成
- 例：「樹勢2、樹形3。葉量の減少および多数の枯れ枝が認められる。これらの所見から、樹勢はやや不良、樹形は不良と判断した。」

**実装場所**：`src/lib/generateJudgmentReason.js`

#### e) 写真アノテーション（v2.8）

全景写真の上に病害マーク（赤矢印＋白背景テキストボックス）を配置できる。

**操作**：
1. 「✏ マーク追加」ボタンでマークモードへ
2. 写真の患部をタップ → モーダル表示
3. チップ選択 or 手入力で病害名確定
4. 患部に赤丸＋テキストボックスが配置される
5. テキストボックスはドラッグで微調整可
6. テキストボックスタップで再編集／削除

**データ保存**：座標は0-1正規化（写真サイズ依存しない）

**実装場所**：
- `src/components/PhotoAnnotator.jsx`
- `src/components/PhotoMarkLabelModal.jsx`

#### f) エクスポート

- **JSON**：構造化データ（写真除く）
- **ZIP**：JSON + 写真ファイル群（カルテ生成ツールへの入力）
- **Excel（シンプル版）**：1シートに全樹データを並べた一覧
- **テキストコピー**：クリップボード経由

### 5.3 オフライン対応

- IndexedDBに全データ保存（idb-keyval）
- vite-plugin-pwaでService Worker登録
- 一度ロードすれば電波なしでも動作
- iPhoneにPWAインストール可能（ホーム画面追加）

## 6. カルテ生成ツール仕様

### 6.1 役割

PWAから出力されたJSON+写真ZIPを入力に、行政提出用カルテExcelを生成する。

### 6.2 構成

```
karte-generator/
├── generate.py              # コアロジック（CUI実行可）
├── generate_gui.py          # tkinter GUIアプリ
├── photo_annotator.py       # 写真アノテーション焼き込み
├── カルテ生成.bat           # ダブルクリック起動用
├── build.bat                # PyInstaller exe化
├── karte_generator.spec     # PyInstaller 設定
├── templates/
│   ├── shibuya.xlsx         # 渋谷氷川の杜様式
│   └── shibuya.json         # セルマッピング定義
└── README.md
```

### 6.3 テンプレート方式

行政によってカルテのフォーマットが異なるため、テンプレート機構を採用：

- `templates/{name}.xlsx` ：行政別の様式Excel
- `templates/{name}.json` ：JSONフィールド → Excelセルアドレスのマッピング定義

新様式に対応するには、テンプレートExcelとマッピングJSONを `templates/` に追加するだけ。

### 6.4 写真アノテーション焼き込み（v2.8.1）

PWA で記録された `annotations` 配列をPILで写真に焼き込む：

1. 正規化座標（0-1）を写真の実ピクセルに変換
2. 患部に赤丸（ellipse）
3. テキストボックスから患部へ赤矢印（line + 三角形のpolygon）
4. 矢印の起点はテキストボックスの境界（中心ではない）
5. 白背景＋赤枠の矩形 + 黒文字
6. 写真サイズに応じて文字サイズ・線太さを自動スケール

**実装場所**：`karte-generator/photo_annotator.py`

### 6.5 GUIアプリ

`generate_gui.py`：
- ドラッグ&ドロップ（tkinterdnd2）でJSONを受け取る
- 進捗バー表示
- 複数JSONを統合可能（事務所メンバー分担運用）
- 完了後フォルダを自動オープン

### 6.6 配布

PyInstallerで`.exe`化して、Pythonがインストールされていない環境にも配布可能（同業者向け）。
**現状の課題**：exe化したときに`templates/`の相対パスが解決できない問題が未解決（要修正：`sys.frozen`分岐）。

## 9. 既知の制約・今後の課題

### 緊急度高め

#### 9.1 写真アノテーションのテキストボックス見切れ

**症状**：座標が画像範囲を超えるとテキストが切れる。
**原因仮説**：PWA側のドラッグ時のboundary clamp が甘い、もしくは PIL 描画時の boundary check が無い。
**実装場所**：`src/components/PhotoAnnotator.jsx`、`karte-generator/photo_annotator.py`

#### 9.2 PyInstaller exe化のテンプレートパス問題

**症状**：`.exe`実行時に`templates/`が見つからずエラー。
**原因**：`__file__`が一時展開フォルダを指す。
**解決方針（既知・未実装）**：

```python
if getattr(sys, 'frozen', False):
    SCRIPT_DIR = Path(sys.executable).parent
else:
    SCRIPT_DIR = Path(__file__).parent
```

### 緊急度中

#### 9.3 診断文生成の文章品質

現在は辞書＋テンプレート方式。**カルテ調**にはなったが、樹木医の業務文書としての完成度は改善余地あり。  
将来的にClaude APIで生成する選択肢もあるが、オフライン要件と料金面の懸念。

#### 9.4 Vercel Hobby商用利用問題

Vercel HobbyプランはToS上の商用利用NG。同業者数人で使うとグレーゾーン。
**移行先候補**：Cloudflare Pages（無料・商用OK・帯域無制限）。

### 緊急度低

#### 9.5 データ共有機能（保留中）

事務所2-3人で「現場分担型」運用したいニーズ。
現状はJSONエクスポート → Dropboxで共有 → リーダーがPC側で統合の運用。  
将来的にクラウドバックアップ → チーム同期へ段階的に拡張可能。

#### 9.6 辞書編集UI

診断文生成のキーワード辞書をUIで編集できるようにする案。  
同業者に配布したときに語彙差を吸収するため。  
現状はソース直編集。
